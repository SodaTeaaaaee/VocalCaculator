## 1. 目标摘要

构建一个保留模式、原生 GUI、跨平台的语音计算器应用：

1. 主支持平台：Windows 与 Android
2. 次优先级平台：Linux、macOS、iOS
3. 前端技术栈：避免 Web 技术栈
4. UI 类型：原生 retained-mode GUI，不使用游戏引擎
5. 业务核心：老式即时执行型四则运算计算器
6. 音频模式：静音、正常播报、损坏播报、音乐模式

产品目标不是“做一个能算的 Demo”，而是“做一个可维护、可测试、可继续发布的应用骨架，并尽可能把 V1 功能做完整”。

## 2. 已冻结的技术决策

除非出现经过联网核实的一手资料证明当前方案不可行，否则不得擅自推翻以下决策。

### 2.1 GUI 栈

必须使用 `Slint 1.16.x` 系列作为 GUI 主栈。

理由：

1. 需要原生 retained-mode GUI
2. 需要 Windows + Android 官方支持
3. 需要 Rust 原生集成
4. 需要 declarative UI、状态与动画能力

禁止引入：

1. Tauri
2. Electron
3. Iced 作为主栈
4. egui 作为主栈
5. Bevy 或其他游戏引擎作为主 UI 栈
6. 第二套 GUI 栈

### 2.2 音频栈

必须使用 `kira 0.12.x` 作为唯一高层播放 API。

禁止在应用层直接使用：

1. `rodio`
2. `cpal`
3. `oddio`
4. `SDL2_mixer`

说明：

1. `cpal` 可以作为 `kira` 的后端依赖存在，但应用代码不得直接围绕 `cpal` 写业务逻辑
2. 所有短音频都必须走 `StaticSoundData`
3. 所有交互音频都必须可立即打断

### 2.3 数值栈

计算器核心数值必须使用 `rust_decimal 1.42.x` 或同系列兼容版本。

禁止使用：

1. `f32`
2. `f64`
3. 任意把显示值先转成浮点再运算的实现

原因：

1. 这是计算器，不是近似数值演示器
2. 百分比、记忆、MU、链式等行为应避免二进制浮点舍入噪声

### 2.4 序列化与配置

配置必须使用：

1. `serde 1.x`
2. `toml 1.x`

配置与应用数据路径必须使用：

1. `sysdirs 0.9.x`

说明：

1. `sysdirs` 明确支持 Windows 与 Android
2. Android 必须使用自动初始化能力或显式初始化

### 2.4.1 依赖复用规则

`Cargo.toml` 当前已经钉死一批核心依赖，执行模型必须优先复用，不得盲目新增：

1. `slint`
2. `slint-build`
3. `kira`
4. `rust_decimal`
5. `rust_decimal_macros`
6. `serde`
7. `toml`
8. `sysdirs`
9. `log`
10. `env_logger`
11. `android_logger`
12. `thiserror`
13. `anyhow`
14. `proptest`
15. `rstest`

新增依赖前必须满足：

1. 先说明现有依赖为何不足
2. 先联网核实该依赖的官方文档与许可
3. 先确认该依赖不会引入第二套 GUI、第二套音频主栈或第二套配置栈

若没有充分理由，默认禁止新增依赖。

### 2.5 Android 构建

Android 构建必须使用：

1. `cargo-apk`
2. `slint::android`

禁止引入 Kotlin/Java UI 层来重写主界面。

允许最小 Android 包装层仅用于：

1. 清单配置
2. 签名配置
3. 运行入口

## 3. 关键法律与分发边界

本节是硬约束。
不得以“作者应该不在乎”或“看起来像公共领域”为由绕过。

### 3.1 Slint 许可

用户已明确接受 Slint 社区许可要求的归因。

因此执行模型必须：

1. 在应用内提供可从顶层访问的 About 页面，显示 Slint 归因
2. 或在公开下载页显示 Slint 归因徽标
3. 两者同时做更好

### 3.2 AR7778 外部仓库

外部仓库：

`https://github.com/evnchn-AR7778/AR7778-digitized-MIDI`

截至 2026-06-08，未发现明确 license。
因此不能把该仓库内容默认视为可自由再分发。

所以必须采用以下策略：

1. 允许工程上支持该仓库内容
2. 允许本地开发或内部验证使用该仓库内容
3. 不得把该仓库资产默认写入“可安全公开再分发”的结论
4. 不得在无进一步人工确认前，把其二进制资产直接当作公开发布的默认内置资产

### 3.3 音乐模式的法律安全策略

音乐模式必须保留，但实现方式必须尽量克制：

1. 不依赖上游仓库中的歌曲 MIDI 作为产品必需资源
2. 优先只使用上游的 `256.sf2` 作为开发期音色来源
3. 为按钮音和播报 token 生成我们自己的单音符或短音型资源
4. 不使用上游歌曲作为运行时播放内容

必须把音乐模式拆成两个层级：

1. 工程支持层：代码、映射、生成脚本、缺省禁用逻辑全部实现
2. 公开分发层：只有在人工确认授权后，才允许把基于上游资源生成的音色包随应用一起发布

因此公开可分发版本必须满足下列之一：

1. 音乐模式默认禁用，等待用户手工安装外部资产包
2. 使用已经人工确认可再分发的替代音色包

## 5. 架构总则

### 5.1 单一真相源

Rust 业务层是唯一真相源。

UI 不得自行保存第二份业务状态。

### 5.2 绝对禁止的坏实现

1. 在 UI 事件回调里直接写完整计算逻辑
2. 用浮点数表示核心计算状态
3. 每次按钮按下时从磁盘读取或解码 WAV
4. 运行时用模糊文件名匹配推断损坏音频语义
5. 通过重建整个音频系统来中断单个播放
6. 为了“简单”而把桌面和 Android 写成两套不同业务实现

### 5.3 推荐代码组织

优先采用单包多目标，而不是一开始就做大 workspace。

目标结构：

1. `src/main.rs`：桌面入口
2. `src/lib.rs`：共享入口与 Android `android_main`
3. `src/app/`：应用状态与用例
4. `src/core/`：纯计算逻辑、数字格式化、token 生成
5. `src/audio/`：`kira` 封装、资源注册表、模式策略
6. `src/ui/`：Rust 到 Slint 的桥接
7. `ui/`：`.slint` 文件
8. `build.rs`：Slint 编译与可能的资源生成

后续只有在结构已经被证明失控时，才允许升级为 workspace。

## 6. UI 规格

### 6.1 总体要求

应用仍然必须看起来像计算器，而不是“通用桌面表单”。

必须做到：

1. 主界面是计算器式按键布局
2. 显示区突出、右对齐
3. 记忆键与主运算区有清晰分层
4. 键盘触控和鼠标点击都舒服
5. Android 上单手触控可用

### 6.2 Slint 风格策略

1. 默认使用 `native` 风格
2. Windows 实际呈现应接近 `fluent`
3. Android 实际呈现应接近 `material`
4. 主键盘不得简单堆叠标准按钮了事，必须做定制外观
5. 可使用 `Palette` 与 `StyleMetrics` 保持平台一致性

### 6.3 布局要求

基础布局：

1. 显示屏
2. 记忆键一行
3. 主键盘四列
4. 模式切换入口
5. 可访问的 About/Settings 入口

必须至少包含：

1. `MC`
2. `MR`
3. `M-`
4. `M+`
5. `AC`
6. `C`
7. `⌫`
8. `÷`
9. `7 8 9 ×`
10. `4 5 6 -`
11. `1 2 3 +`
12. `± 0 . =`
13. `%`
14. `√`
15. `MU`

### 6.4 动画要求

必须有：

1. 按下反馈
2. 模式切换反馈
3. 错误态视觉反馈

禁止：

1. 夸张的游戏化粒子效果
2. 高频持续动画
3. 影响计算输入节奏的过长过渡

### 6.5 窗口与移动端策略

Windows：

1. 支持缩放
2. 设置合理最小尺寸

Android：

1. V1 只强制支持竖屏
2. 横屏可后续处理
3. 触控目标尺寸优先，不要先追求极小边框

## 7. 计算逻辑规格

### 7.1 核心模式

这是老式即时执行型计算器，不是表达式解析器。

必须支持：

1. 运算符即时执行
2. 无优先级
3. 连续 `=` 重复最后一次运算
4. 结果后数字覆盖输入

### 7.2 百分比规则

百分比必须按老式通用计算器语义处理：

1. `A + B%` => `A + A * B / 100`
2. `A - B%` => `A - A * B / 100`
3. `A × B%` => `A * B / 100`
4. `A ÷ B%` => `A / (B / 100)`
5. 单独 `B%` => `B / 100`

### 7.3 MU 规则

`MU` 必须按 CASIO 常见 markup 语义实现：

1. `cost MU rate %` => `selling_price = cost / (1 - rate / 100)`
2. 在典型序列 `120 MU 25 %` 中，结果应为 `160`
3. 随后按 `=` 应得到利润额 `40`

### 7.4 其他键规则

1. `AC`：清空输入、运算上下文、错误态、记忆
2. `C`：清空当前输入，不清空记忆
3. `⌫`：删除当前输入最后一位
4. `±`：切换当前输入符号
5. `√`：对当前显示值开方；负值进入错误态
6. `MR`：回忆记忆值到当前显示
7. `M+` / `M-`：基于当前显示修改记忆值
8. `MC`：清除记忆

### 7.5 错误态

至少覆盖：

1. 除以零
2. 负数开方
3. 非法百分比上下文
4. 不可表示结果

错误态必须：

1. 视觉可见
2. 音频可见
3. 限制后续输入直到用户恢复

## 8. 语义 token 流

实现时必须把“计算结果/按键事件”先转成语义 token，再由不同音频模式消费。

禁止直接在每个按钮分支里手写音频播放。

示例 token：

1. `DigitZero`
2. `DigitOne`
3. `UnitTen`
4. `OperatorAdd`
5. `ActionEquals`
6. `ActionError`
7. `SignNegative`
8. `FunctionPercent`

这样做的目的是：

1. 正常模式消费语音 token
2. 损坏模式消费相同语义 token，但替换为 broken/noise 策略
3. 音乐模式消费相同语义 token，但替换为音符/动机

## 9. 正常播报模式

### 9.1 基础原则

1. 按键按下时播报按键语义
2. 结果出现时按中文数词规则播报结果
3. 错误时播报错误

### 9.2 数字播报算法

必须支持：

1. 正数
2. 负数
3. 小数
4. 零插入规则

建议保证自然播报范围至少覆盖整数部分绝对值 `< 10^12`。
超过该范围时，不得崩溃；可以降级为：

1. 截断为不可播报并仅显示
2. 或进入定义好的“结果过大，不播报”策略

### 9.3 结果播报优先级

结果播报的正确性优先于“每次都播报”。

宁可在无法自然表达时跳过结果播报，也不要播错。

## 10. 损坏播报模式

### 10.1 不得使用运行时模糊推断

不得在运行时根据文件名模糊匹配。

必须做法：

1. 把正常与损坏资产语义显式登记到受版本控制的映射表
2. 运行时只读取映射表

### 10.2 播放策略

对每个语义 token：

1. 如果有 broken 版本，则在 normal pool 与 broken pool 之间交替
2. 如果没有 broken 版本，则在 normal pool 与全局 noise pool 之间交替
3. 绝不允许把一个 token 播成另一个 token 的“正确语义”

### 10.3 不确定文件

以下文件在没有人工再确认前，默认归入噪声池：

1. `59_0xDD_U♂.wav`
2. `61_0xE4_U♂.wav`
3. `62_0xE7_于.wav`
4. `67_0xF8_卟零.wav`
5. `68_0xFF_卟零♂.wav`

如果后续通过人工确认这些文件确实应映射到某个语义 token，必须更新映射表与说明文档。

## 11. 音乐模式

### 11.1 产品语义

音乐模式不是“切到另一首歌播放器”。
它仍然是计算器，只是把语音 token 换成音乐 token。

### 11.2 必须保持的 UI 行为

1. 界面不变
2. 按键逻辑不变
3. 只有发声方式变化

### 11.3 音乐模式实现策略

音乐模式必须复用正常模式的语义 token 流。

必须做到：

1. 数字 token 映射为音高
2. 运算符 token 映射为低音或短动机
3. 结果 token 流映射为音符序列
4. 错误 token 映射为不和谐提示音

### 11.4 音色来源策略

优先策略：

1. 使用上游 `256.sf2`
2. 用本仓库自生成的单音符 MIDI 模板渲染所需 note bank
3. 运行时只播放预生成 WAV，不运行 MIDI 合成器

禁止：

1. 运行时调用外部 `fluidsynth`
2. 在 Android 上依赖桌面 MIDI 工具
3. 把上游歌曲 MIDI 当作产品功能资源

### 11.5 缺省行为

如果未检测到合法可用的音乐资产包：

1. 应用仍必须正常运行
2. 音乐模式入口应显示为禁用或不可用
3. 必须告诉用户缺失的是外部资产包，而不是程序故障

## 12. 资产策略

### 12.1 本仓库内置语音 WAV

`resource/Vocal/Normal` 与 `resource/Vocal/Broken` 中的短 WAV 必须在编译期嵌入或在启动时一次性载入到内存。

优先方案：

1. 编译期 `include_bytes!`
2. 通过 `StaticSoundData::from_cursor` 解码

这样做的原因：

1. 避免 Android APK 资产路径问题
2. 避免运行时文件 I/O
3. 避免交互延迟

### 12.2 外部音乐资产

外部音乐资产不得默认硬编码进公共发布包。

必须使用以下其中一种路径：

1. 构建时检测本地外部资产目录并生成本地音色包
2. 运行时从用户指定目录加载本地音色包

### 12.3 受控映射文件

必须创建显式映射文件，例如：

1. `resource/metadata/normal_tokens.toml`
2. `resource/metadata/broken_tokens.toml`
3. `resource/metadata/music_tokens.toml`

映射文件必须是唯一运行时真相源。

## 13. 播放控制要求

1. 每次播放都持有独立 handle
2. 允许 hard-cut
3. 允许短淡出
4. 快速连续按键时，旧音必须可立即被新音打断
5. 不得通过重建 `AudioManager` 来打断单个 clip
6. 必须记录底层 stream error

## 14. 开发顺序

执行模型必须按以下顺序推进。
不得一上来就同时碰所有模块。

在第 1 步之前，必须先检查项目本地环境；若缺失则先运行 `.local/bootstrap.ps1` 完成本地初始化，不得先尝试自行安装全局工具。

1. 先重建 Cargo 结构与 Slint 最小窗口
2. 再完成纯计算核心与测试
3. 再把计算核心接到桌面 UI
4. 再接正常音频模式
5. 再接损坏模式
6. 再接 Android 构建
7. 最后接音乐模式支持层

如果桌面核心逻辑未稳定，不得提前深入 Android UI 细节。

## 15. 强制子 agent 协议

必须遵守 `.agents/vocal_calculator_subagent_protocol.md`。

最少要求：

1. 非平凡阶段开始前必须先并发派出多个子 agent
2. 每个子 agent 只做一个边界清晰的任务
3. 每个子 agent 都要把结论同步到 `.agents/sync/`
4. 主 agent 必须整合，不得把原始分歧悄悄忽略

禁止修改或新增与本项目无关的目录。

## 17. 验证命令

首选统一验证入口：

`pwsh .agents/verify_vocal_calculator.ps1`

该脚本内部必须依赖项目本地环境，而不是系统全局环境。

该脚本应至少覆盖：

1. `cargo fmt --all -- --check`
2. `cargo clippy --all-targets --all-features -- -D warnings`
3. `cargo test --all-targets --all-features`
4. `cargo doc --no-deps`
5. `cargo build --release`
6. 如果 Android 环境齐全，则执行 `cargo apk build --lib --target aarch64-linux-android`

## 18. 交付物

完成时磁盘上至少必须有：

1. 可运行桌面应用
2. 可构建 Android 包
3. 纯计算核心测试
4. 音频映射元数据文件
5. About/归因页面
6. 第三方说明文档
7. `.agents/sync/` 下的研究同步记录
8. 构建与运行说明

## 19. 停止条件

遇到以下情况必须停止并明确报告，而不是继续乱猜：

1. Android SDK/NDK 或 Java 环境缺失
2. Slint 社区许可归因实现路径不清
3. 外部音乐资产路径或 commit 不明确
4. 同一编译或构建错误连续三次无实质进展
5. 需要用户提供签名密钥、授权、外部二进制工具但当前不存在

## 20. 完成判定

只有同时满足以下条件，才可判定本目标完成：

1. 桌面应用可运行
2. 计算逻辑行为与本文档一致
3. 正常与损坏模式可工作
4. 音乐模式支持层完成，且在有本地资产包时可工作，缺失资产包时可正确降级
5. Windows 构建通过
6. Android 构建通过
7. 验证脚本通过或明确说明哪些门槛因环境前置条件而被阻塞
8. 文档、归因、第三方说明齐全
